## BI 报表平均播放精度修复
- 问题：`GET /admin/bi/zone-stats` 返回的 `avg_plays_per_video` 是原始 float64 除法结果，前端 `fmtNum` 对 <10000 的值直接用 `String(n)` 输出，导致显示 `123.45678901234` 这样的高精度小数。
- 修复：

## BI 报表前后端对齐修复（权限门控+字段不匹配）

**用户反馈**：BI 统计报表没有真实数据和 API，前端有虚假的表和按钮。

**根因**：
1. **权限门控**：所有 BI 路由被 `RequirePermission(dashboard.export)` 拦截，admin 若未配置此权限则全部 403
2. **saveReport 字段不匹配**：前端发 `{name, type, config}`，后端 `name` 和 `query_config` 均为 required
3. **savedReports type 列**：后端返回 `query_config`/`chart_type`，前端用不存在的 `type`
4. **Zone SVG**：`z.zone_id` 不存在于 API response
5. **Creator stats**：后端一次只返回单个维度，前端表格却展示 5 列

**修复**：
- `router.go`：BI 只读 GET 移至公开 admin 组（无需特殊权限），仅 POST/DELETE 保留权限门控
- `admin_bi.go`：重写 `AdminGetCreatorStats` 一次返回全部字段（total_plays/total_coins/fans_count/video_count/article_count）
- `BIReport.vue`：saveReport 字段名修正、savedReports 列绑定 `chart_type`、zone SVG key 改用 index、creator 数据映射、列排序属性全部静态绑定

编译验证：Go build + Vue build 均通过。

**16:09 补充**：用户反馈前端仍无数据。排查发现 Go 编译后未重启后端进程，旧二进制（PID 16504）仍在运行。旧后端返回的单维度 creator 格式与已修改的前端不匹配。
- 操作：`taskkill` 旧进程 → 重启新 `mini-bili.exe`（PID 13328）
- 验证：5 个 API 全量通过 agent-browser 截图确认，creator/zone/timeseries/manuscript/engagement 均展示真实数据
- 结论：不是"虚假的表和按钮"，而是新旧二进制不匹配导致前端 HMR 已更新但后端未更新
